# Contributing to agents-runtime

Thank you for your interest in contributing! This document explains how to get started.

---

## Development Setup

```bash
git clone https://github.com/ahvcxa/agents-runtime.git
cd agents-runtime
npm install
npm test        # Run the test suite (308 tests, ~2s)
```

For detailed setup instructions, see [SETUP_GUIDE.md](SETUP_GUIDE.md).

---

## What Gets Generated After Install

When you run `npm install`, several files are **automatically generated** and should **NOT be committed to git**:

| File/Folder | Generated By | Purpose | Should Commit? |
|---|---|---|---|
| `.agents/manifest.json` | `npm install` (from template) | Merged skill registry | ✗ No |
| `.agents/settings.json` | `npm install` (from template) | Runtime configuration | ✗ No |
| `.agents/memory/` | First skill run | Memory indexes and cache | ✗ No |
| `.agents/logs/` | Skill execution | Execution audit trail | ✗ No |
| `agent.yaml` | `npm run setup` | Instance-specific config | ✗ No |
| `node_modules/` | `npm install` | Dependencies | ✗ No |

**Why not?**
- These are instance-specific or generated at runtime
- Each developer will have different values
- Committing them creates merge conflicts and bloat
- They're in `.gitignore` by design

## Template vs. Generated Files

### Template Files (Commit These)

```
template/.agents/              # Distributed templates
├── manifest.json              # Skill registry template
├── settings.json              # Default settings
├── helpers/                   # Utility modules
├── hooks/                     # Lifecycle hooks
└── [skill definitions]        # Skill code
```

**These ARE version-controlled** because they're part of the distribution package.

### Generated Files (Don't Commit These)

```
.agents/                       # Runtime-generated
├── manifest.json              # (Merged from template)
├── settings.json              # (Generated)
├── memory/                    # (Runtime data)
└── logs/                      # (Execution logs)
```

**These are gitignored** because they're local to each developer's machine.

## Project Structure

| Path | Purpose |
|------|---------|
| `bin/agents.js` | CLI entry point |
| `src/engine.js` | Central orchestrator |
| `src/agent-runner.js` | Skill lifecycle pipeline |
| `src/analyzers/` | Language-specific analyzers (JS, Python) |
| `src/registry/` | Hook and skill registries |
| `src/memory/` | ACL-backed memory store |
| `tests/` | Jest test suites + isolated fixture project |
| `template/.agents/` | Skill templates (distributed with package) |

The `template/.agents/` directory contains the templates that are installed into user projects. The `.agents/` directory in the root is generated at runtime and should never be committed.

## Contributing a New Language Analyzer

1. Create `src/analyzers/<lang>-analyzer.js`
2. Export two functions:
   ```javascript
   module.exports = {
     analyzeCode<Lang>(lines, relPath) { return findings[]; },
     auditSecurity<Lang>(lines, relPath) { return findings[]; },
   };
   ```
3. Copy the analyzer to `template/.agents/helpers/<lang>-analyzer.js` (NOT `.agents/`)
4. Wire it into the skill handlers (dispatch by file extension)
5. Add tests in `tests/`

---

## Contributing a New Skill

1. Create `template/.agents/skills/<skill-name>/SKILL.md` with the required YAML frontmatter
2. Create `template/.agents/skills/<skill-name>/handler.js` implementing `async execute(ctx)`
3. Register the skill in `template/.agents/manifest.json` (NOT `.agents/manifest.json`)
4. Update `README.md` with the skill description

**IMPORTANT:** Always edit files in `template/` directory, not the root `.agents/` directory. The root `.agents/` is auto-generated and your changes will be lost.

---

## Code Style

- **No external runtime dependencies** beyond `commander`, `gray-matter`, `js-yaml`
- All new analyzer logic must be pure Node.js (no native bindings)
- Every exported function must have a JSDoc comment
- Finding objects must conform to the `Finding` interface in `code-analysis/SKILL.md`

---

## Pull Request Guidelines

- One feature / bug fix per PR
- Include tests for new functionality
- Update `README.md` if CLI commands or skill behavior changes
- Ensure `npm test` passes with no failures
- Never commit generated files (`.agents/`, `agent.yaml`, `node_modules/`)
- Always edit `template/.agents/` for skill changes, not `.agents/`

---

## Reporting Issues

Use the GitHub issue templates:
- 🐛 **Bug Report** — for incorrect findings, crashes, or unexpected behavior
- 💡 **Feature Request** — for new skills, language support, or CLI commands
- 🔒 **Security** — for vulnerabilities, please email directly (do not open a public issue)
